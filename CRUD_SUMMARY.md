## How It Works

### Command Structure

```bash
# Display available fields
python3 CRUD.py schema

# Read fields
python3 CRUD.py read: field1, field2, field3

# Create record
python3 CRUD.py create
# or pipe JSON
echo '{"field": "value"}' | python3 CRUD.py create

# Delete records
python3 CRUD.py delete: field=value
```

### Intelligent Database Routing

The system automatically determines where to query based on schema_map.json:

```python
# Example: "read: username, device_model, temperature_c"

username → db: "BOTH" 
  └─ Query both SQL and MongoDB

device_model → db: "SQL"
  └─ Query only SQL database

temperature_c → db: "MONGO"
  └─ Query only MongoDB
  
# Example: "delete: username=john_doe"

username → db: "BOTH"
  └─ Delete from both SQL (structured_data) and MongoDB (unstructured_data)
```

### Field Storage Categories

From analysis of schema_map.json:

| Category | Count | Examples |
|----------|-------|----------|
| SQL Only | 3 | device_model, spo2, purchase_value |
| MongoDB Only | 50+ | temperature_c, ip_address, heart_rate, ... |
| Both Databases | 3 | username, timestamp, sys_ingested_at |

## Integration with Existing System

### Database Handlers
- **SQL**: Uses `mysql.connector` directly (same as sql_handler.py)
- **MongoDB**: Uses `pymongo` directly (same as mongo_handler.py)
- **Reason**: Independent connection lifecycle for CRUD operations

### Schema Integration
- **Source**: `metadata/schema_map.json`
- **Usage**: Determines field storage locations
- **Generated by**: `core/analyzer.py` during streaming operations

## Key Design Decisions

### 1. Independent Connections
CRUD.py creates independent database connections rather than reusing SQLHandler/MongoHandler:
- **Reason**: Different lifecycle needs (streaming vs. interactive)
- **Benefit**: Cleaner separation of concerns

### 2. Schema Metadata Lookup
Uses schema_map.json for field routing:
- **Reason**: Single source of truth for field storage
- **Benefit**: Automatically stays in sync with analyzer output

### 3. Combined Output
Results from both databases merged into single output.txt:
- **Reason**: User wants unified view of data
- **Benefit**: Easy to search and analyze results together

### 4. 1000 Record Limit
Read operations limit to 1000 records per database:
- **Reason**: Memory efficiency, typical use case
- **Benefit**: Fast response times

### 5. Automatic Timestamping
sys_ingested_at added to all CREATE operations:
- **Reason**: Audit trail and data lineage
- **Benefit**: Track when records were created

## How Schema_Map.json Is Used

The schema_map.json file contains the "source of truth" for field storage:

```python
# CRUD.py loads it:
schema_map = {
    "field_name": {
        "db": "SQL|MONGO|BOTH",  # ← Key! Determines routing
        "count": 86950,           # ← Number of records
        "types": ["str"],         # ← Data types
        "is_nested": false,       # ← Structure info
        ...
    }
}

# For each field in a read command:
location = schema_map[field]["db"]
if location in ["SQL", "BOTH"]: query_sql()
if location in ["MONGO", "BOTH"]: query_mongo()
```

This makes CRUD.py automatically adapt to schema changes!

## Error Handling

| Scenario | Handling |
|----------|----------|
| Field not in schema | ValueError with helpful message |
| Database connection fails | Warning, marks as unavailable |
| Invalid SQL query | MySQL error caught and reported |
| Invalid MongoDB query | PyMongo error caught and reported |
| Invalid JSON input | JSONDecodeError caught and reported |
| No results found | Empty results marked in output |
